## 1. 客户业务（来自 wms_v1.docx）

### 1.1 客户工作流程

> 以下为 wms_v1 原文整理的业务流转，是目前唯一的流程输入。

| 环节 | 现状描述 |
|---|---|
| **销售** | 销售成交客户后下单给仓库。下单形式为**纸质手写单 + 电子备份**，呈现为统一格式的下单文档，细项包含：公司名、下单人、日期、合同编号、型号、数量。常规物料不直接发微信给仓库和采购；偶尔定制物料可能发信息（临时通知，后续仍要补手写单 + 电子备份）。 |
| **仓储** | 仓库收到销售下单后，统计物料（定制所需零件等），缺货则手写单给采购。仓库为 **2000 平方米平面库**，仓储需求 **500 万个物料/年**，零配件与成品出入库计件方式为**人工计件（人工数、称重）**。 |
| **采购** | 按仓库手写单进行采购。 |

**关键业务特征：**
- 单据流转依赖纸质 + 电子备份，未智能化。
- 销售下单文档字段固定：公司名、下单人、日期、合同编号、型号、数量。
- 仓库承担"统计物料 + 缺货识别 + 触发采购"的核心枢纽角色。
- 物料规模大（500 万/年），人工计件易错。

### 1.2 项目发起原因（5 条痛点）

1. **库存准确率不足**：频繁账实不符，缺料导致生产停工，生产周期预期外延长。
2. **生产领料耗时久**：错领漏领导致工期延长。
3. **销售与仓库信息同步不及时**：响应滞后，导致排单失序。
4. **盘点耗时费力**：循环盘点难以执行。
5. **余料管理欠缺**：影响成本核算准确性，导致物料浪费与新品开发时协同设计困难。

### 1.3 项目达成目标（3 条）

1. **提升库存准确率**，达成订单和物料的动态匹配。
2. **提升物料出入库效率**，及时完成准确采购和物料追踪，克服订单制作周期中预期外少件、缺件导致的拖延工期与制造低效。
3. **实现成品全生命周期追溯**（采购追踪；成品追踪——人工输入）。

### 1.4 规模与约束

| 项 | 值 / 说明 |
|---|---|
| 仓库面积 | 约 2000 平方米平面库 |
| 物料规模 | 约 500 万个物料/年 |
| 计件方式 | 人工计件（人工数、称重） |
| 成品追溯 | 人工输入（非自动采集） |
| 单据形式 | 纸质手写单 + 电子备份 |

## 2. 当前系统现状

### 2.1 技术栈

| 层 | 技术 |
|---|---|
| 前端 | Vue 3 + Vite + Element Plus + Pinia + Vue Router + Axios + ECharts |
| 后端 | Spring Boot 3.3.5 + MyBatis-Plus 3.5.5 + Sa-Token 1.37.0 |
| 数据库 | MySQL 8.0 |
| 运行环境 | JDK 17、Node.js 18 |
| 接口规范 | 统一 `{code,msg,data}`，上下文前缀 `/api`，鉴权 `Authorization: Bearer token` |
| 文档 | OpenAPI + Knife4j（`/api/doc.html`） |

### 2.2 当前已具备的能力

- **认证授权**：Sa-Token 登录态，Session 写入 `role/deptId/deptCode`；`AuthzService` 统一权限抽象。
- **组织与权限**：`superadmin / admin / employee` 三角色 + `dept_id/dept_code` 双维；5 部门（财务/销售/仓储/采购/人事）。
- **基础资料**：供应商（`base_supplier`）、商品（`base_goods`）。
- **业务单据**：采购、采购退货、销售、销售退货（`biz_*`）；当天可删、历史走作废/红冲审批。
- **审批流**：采购/销售管理员提交历史作废申请 → 仓储管理员审批执行（`biz_approval_order`）。
- **库存预警**：低库存/零库存预警中心，仓储/采购/销售共享只读。
- **工作要求闭环**：发布 → 接受/拒收 → 执行 → 提交 → 审核；含超时独立维度建模与站内消息提醒。
- **统计图表**：销售统计、毛利分析（财务管理员专属）。
- **站内消息中心**：所有登录用户可用，工作要求超时消息统一收口。
- **AI 助手**：项目知识问答 + 多模型路由（通义千问/智谱 GLM/Kimi/DeepSeek）+ 历史会话 + 审计留痕。
- **安全审计**：登录日志、操作日志、错误日志、IP 策略。

### 2.3 当前数据库表（22 张）

```
组织权限   sys_user / sys_dept / sys_employee / sys_notice / sys_message
审计治理   sys_error_log / sys_ip_policy / sys_login_log / sys_operation_log
基础资料   base_supplier / base_goods
业务单据   biz_purchase / biz_purchase_return / biz_sales / biz_sales_return
审批       biz_approval_order
工作要求   work_requirement / work_requirement_assign / work_requirement_attachment
AI 助手    ai_conversation / ai_message / ai_model_call_log
```

### 2.4 当前角色 × 菜单矩阵

| 角色 | 主要菜单 |
|---|---|
| superadmin | 首页、公告管理、用户管理、超管总览、AI 助手、安全审计 |
| purchase admin | 首页、商品进货、进货退货、预警中心、通知、AI 助手、用户部门管理 |
| sales admin | 首页、商品销售、销售退货、预警中心、通知、AI 助手、用户部门管理 |
| warehouse admin | 首页、供应商管理、商品资料管理、预警中心、作废审批、通知、AI 助手、用户部门管理 |
| hr / finance admin | 首页、部门/员工管理或销售统计、通知、AI 助手、用户部门管理 |
| employee | 单页工作台：档案、联系方式、部门联络、工作要求、公告、提醒、AI 助手入口 |

### 2.5 当前业务链路（与客户流程对照）

当前系统已有的链路：
- **采购链路**：进货单 → 入库增库存（`biz_purchase`，触发存储过程 `sp_purchase_add_stock`）。
- **销售链路**：销售单 → 出库减库存（`biz_sales`）；销售退货回流库存。
- **库存联动**：进销单据直接影响 `base_goods` 库存数量；预警中心基于库存阈值。

> ⚠️ 注意：当前系统**没有"销售下单 → 仓库统计物料 → 缺货触发采购"的协同链路**，销售单直接扣库存，仓库不参与中间统计与缺货识别。

## 3. 差距分析：客户需求 vs 当前系统

| # | 客户诉求（wms_v1） | 当前系统 | 差距 | 优先级 |
|---|---|---|---|---|
| 1 | 销售下单（公司名/下单人/日期/合同编号/型号/数量）流转给仓库 | `biz_sales` 有销售单，但直接扣库存，无"下单→仓库确认"中间态 | 缺销售下单→仓库统计确认环节 | P0 |
| 2 | 仓库统计物料、识别缺货、手写单给采购 | 无仓库统计/缺货识别环节；预警仅有低库存/零库存 | 缺缺货识别与采购触发链路 | P0 |
| 3 | 库存准确率提升、订单与物料动态匹配 | 有库存数量与预警，但无动态匹配、无账实校验机制 | 缺盘点校验、缺料与订单联动 | P0 |
| 4 | 生产领料（错领漏领问题） | **完全无生产领料模块** | 缺生产领料单 + 复核 | P0 |
| 5 | 销售与仓库信息同步、排单不失序 | 销售单直接扣库存，仓库被动 | 缺销售→仓库协同确认流 | P1 |
| 6 | 盘点（循环盘点可执行） | **完全无盘点模块** | 缺盘点单 + 差异处理 | P1 |
| 7 | 余料管理（成本核算 + 新品协同） | **完全无余料管理** | 缺余料/边角料台账 | P1 |
| 8 | 成品全生命周期追溯（采购追踪 + 成品追踪，人工输入） | 无批次/序列号/追溯 | 缺追溯链路 | P1 |
| 9 | 物料出入库效率提升 | 有进销单据，但计件仍靠人工录入 | 缺扫码/复核提效手段（可后期） | P2 |
| 10 | 采购按仓库缺货单采购 | 采购单独立存在，不与缺货联动 | 缺缺货→采购申请联动 | P1 |

### 3.1 核心结论

当前系统是**通用进销存**，业务模型为"进货增库存 / 销售减库存"，缺少客户最核心的三条链路：

1. **销售→仓库→采购的协同链路**（下单确认 / 缺货识别 / 采购触发）。
2. **生产领料链路**（领料 / 补料 / 退料 / 复核）。
3. **库存治理链路**（盘点 / 余料 / 追溯）。

改造应**以新增模块为主，不重写**已验证的认证、权限、审批、消息、AI 助手能力。

## 4. 改造路线图

### 阶段 1：补齐客户主流程（P0）

- [ ] **销售下单协同**：销售单增加"下单→仓库确认"状态，仓库确认后才扣库存；下单文档字段对齐（公司名/下单人/日期/合同编号/型号/数量）。
- [ ] **缺货识别与采购触发**：仓库统计物料时识别缺货，生成采购申请单，流转给采购。
- [ ] **生产领料模块**：领料单（领料/补料/退料）+ 库存扣减 + 复核确认，解决错领漏领。
- [ ] **库存动态匹配**：订单物料需求 vs 当前库存 vs 在途采购的可见性。

### 阶段 2：库存治理与追溯（P1）

- [ ] **盘点模块**：全盘 / 抽盘 / 循环盘点 + 差异处理 + 审批，解决循环盘点难执行。
- [ ] **余料管理**：余料/边角料台账（入库 / 领用 / 报废 / 估值），可下载供新品设计参考。
- [ ] **追溯链路**：采购追踪（进货批次）+ 成品追踪（人工输入序列号/批次），实现全生命周期追溯。
- [ ] **销售→仓库信息同步强化**：站内消息 + 待办提醒，保障排单同步。

### 阶段 3：提效与扩展（P2）

- [ ] 出入库扫码 / 复核预留接口（提升人工计件效率与准确率）。
- [ ] 采购到货预期、在途库存管理。
- [ ] 利用现有 AI 助手沉淀客户操作手册与流程问答。

## 5. 模块改造方案

### 5.1 销售下单协同（P0，对齐 wms_v1 销售环节）

**问题：** wms_v1 明确"销售下单给仓库"，当前系统销售单直接扣库存，仓库不参与。

**方案：**
- `biz_sales` 增加 `confirm_status`（待仓库确认 / 已确认 / 已出库）。
- 销售建单后状态为"待仓库确认"，**不立即扣库存**；仓库管理员确认后才扣库存并出库。
- 下单文档字段对齐 wms_v1：公司名、下单人、日期、合同编号、型号、数量（部分字段可能需扩展 `biz_sales`）。
- 复用现有站内消息：销售建单 → 推送仓库管理员待确认提醒。

**前端：** `views/business/SalesView.vue` 增加"待仓库确认"状态流转；仓库管理员侧增加"销售单确认"入口。

### 5.2 缺货识别与采购触发（P0，对齐 wms_v1 仓储环节）

**问题：** wms_v1 明确"仓库统计物料，缺货手写单给采购"，当前系统无此环节。

**方案：**
- 仓库确认销售单时，系统按订单物料数量 vs 当前库存自动识别缺货，生成**采购申请单**（新增表 `biz_purchase_request`）。
- 采购申请单流转给采购管理员，采购据此创建 `biz_purchase` 进货单。
- 采购到货入库后，关联更新采购申请单状态。

**复用：** 可借鉴"工作要求"的发布→执行→提交→审核闭环模式（申请→采购→到货→入库）。

### 5.3 生产领料模块（P0，对齐痛点 2）

**问题：** wms_v1 痛点"生产领料耗时久，错领漏领导致工期延长"，当前系统完全无此模块。

**方案：**
- 新增 `biz_pick_list`（领料单）：领料类型（领料 / 补料 / 退料）、领料人、物料明细、数量、关联销售单/生产任务。
- 流程：申请 → 仓库发料（扣库存）→ 领料人确认（复核，防错领漏领）。
- 退料回流入库。
- 缺料时接入预警中心。

**前端：** 新增"生产领料"菜单（仓储管理员 + 生产相关角色）。

### 5.4 盘点模块（P1，对齐痛点 4）

**问题：** wms_v1 痛点"盘点耗时费力，循环盘点难以执行"。

**方案：**
- 新增 `biz_stocktake`（盘点单）+ `biz_stocktake_detail`（盘点明细）。
- 类型：全盘 / 抽盘 / 循环盘点。
- 字段：账面数量 / 实盘数量 / 差异 / 处理结果。
- 差异处理审批（复用 `biz_approval_order` 框架）。

### 5.5 余料管理（P1，对齐痛点 5）

**问题：** wms_v1 痛点"余料管理欠缺，影响成本核算与新品协同"。

**方案：**
- 新增 `biz_scrap_material`（余料/边角料台账）：类型（余料 / 边角料）、状态（入库 / 领用 / 报废）、估值单价（人工输入）。
- 余料库存信息**可下载**（wms_v1 明确"供新品开发协同设计"）。

### 5.6 成品全生命周期追溯（P1，对齐目标 3）

**问题：** wms_v1 目标"实现成品全生命周期追溯（采购追踪；成品追踪——人工输入）"。

**方案：**
- **采购追踪**：`biz_purchase` 关联批次号，进货批次 → 物料 → 成品的正向追溯。
- **成品追踪**：新增成品序列号/批次记录（**人工输入**，wms_v1 明确非自动采集），绑定其使用的物料批次。
- 新增 `base_batch`（批次）+ `base_serial_number`（序列号，人工输入）。
- 提供按序列号 / 批次反查来源的功能。

### 5.7 前端改造要点

- 新增菜单：生产领料、盘点管理、余料管理、追溯查询（按角色分配）。
- 复用 `utils/bizDocumentState.js` 模式统一新单据状态解释。
- 复用 `utils/permission.js` 的 `v-permission` 指令做按钮级权限。
- 销售单增加"待仓库确认"状态展示。

## 6. 接口规划

沿用 `/api` 前缀 + `{code,msg,data}` 规范，鉴权 `Authorization: Bearer token`，复用 `AuthzService` 做部门边界判断。

```
销售协同      /api/business/sales/{id}/confirm          仓库确认销售单
采购申请      /api/business/purchase-requests/*          缺货触发的采购申请
生产领料      /api/business/pick-lists/*
盘点          /api/business/stocktake/*
余料          /api/business/scrap-material/*
追溯          /api/base/batches/*
              /api/base/serial-numbers/*
              /api/business/trace/{serialOrBatch}        反查来源
```

- 审批类操作复用 `biz_approval_order` 框架。
- 文件上传复用 `/api/upload/image`。

## 8. 风险与注意事项

1. **需求唯一来源单薄**：`wms_v1.docx` 仅约 530 字，只有工作流程、5 痛点、3 目标，**缺少字段级细节、权限要求、编码规则、组织架构**等。开发前需与客户补充确认（见 §9）。
2. **不破坏现有能力**：改造以新增模块为主，保留已验证的认证、权限、审批、消息、AI 助手能力。
3. **追溯人工输入**：wms_v1 明确成品追踪为"人工输入"，本期勿过度设计自动采集，扫码/复核作为未来扩展。
4. **物料规模大**：500 万物料/年，库存表查询与预警性能需关注索引设计。
5. **销售协同改造影响面**：销售单从"直接扣库存"改为"仓库确认后扣库存"，会影响现有销售链路与库存联动逻辑，需谨慎回归测试。
