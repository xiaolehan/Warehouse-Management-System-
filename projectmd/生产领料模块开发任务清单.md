## 0. 模块设计概要

### 0.1 业务流程（对齐 wms_v1）

```
领料人申请（领料/补料/退料）
  → 仓储管理员发料（扣库存 / 退料则回流入库）
    → 领料人确认收货（复核，防错领漏领）
      → 完成
```

- **领料**：从库存扣减物料。
- **补料**：领料不足时再次领料，流程同领料。
- **退料**：领料人退还物料，库存回流入库。
- **复核环节**：发料后领料人需确认，未确认前状态为"已发料"，确认后"已完成"。这是解决"错领漏领"的关键。

### 0.2 状态机

| 状态值 | 名称 | 说明 |
|---|---|---|
| 1 | 待发料 | 领料人已申请，待仓储管理员发料 |
| 2 | 已发料 | 仓储已发料，待领料人确认 |
| 3 | 已完成 | 领料人已确认，闭环 |
| 4 | 已驳回 | 仓储管理员驳回申请（如缺料且无法补货） |

**发料时的库存处理（按 pick_type 区分）：**
- 领料（PICK）/ 补料（SUPPLY）：发料时**扣减**库存（明细逐条 `decreaseStock`，任一缺料则整单失败回滚）。
- 退料（RETURN）：发料时**回流入库**（明细逐条 `increaseStock`）。

> 退料同样走 待发料→已发料→已完成（Q4 决定：退料需仓储确认入库，发料动作即入库，但仍需领料人确认收货闭环）。

### 0.3 权限设计（Q1 确认：限定销售/仓储 admin 申请）

| 操作 | 权限 |
|---|---|
| 申请领料 | **销售 admin / 仓储 admin** 或超管（`requireAnyDeptAdminOrSuperAdmin(DEPT_SALES, DEPT_WAREHOUSE)`） |
| 发料 / 驳回 | 仓储管理员（`DEPT_WAREHOUSE`）或超管 |
| 确认收货 | 申请人本人 |
| 查询列表 | 申请人看本人；仓储管理员看全部；销售 admin 看本部门；超管看全部 |

### 0.4 数据模型（Q2 确认：多商品主从表）

主表 `biz_pick_list`（单据头）：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK auto | |
| pick_no | varchar(32) | 领料单号，`CodeGenerator.pickListNo()` |
| pick_type | varchar(16) | PICK（领料）/ SUPPLY（补料）/ RETURN（退料） |
| source_sales_id | bigint | 关联销售单（Q3 确认：可选） |
| status | tinyint | 1待发料/2已发料/3已完成/4已驳回 |
| applicant_id | bigint | 申请人ID |
| applicant_name | varchar(64) | 申请人姓名 |
| operator_id | bigint | 发料人ID（仓储） |
| operator_name | varchar(64) | 发料人姓名 |
| operation_time | datetime | 发料时间 |
| confirm_time | datetime | 确认收货时间 |
| reject_reason | varchar(255) | 驳回原因 |
| remark | varchar(255) | 备注 |
| create_time | datetime | |
| update_time | datetime | |
| is_deleted | tinyint | 逻辑删除（TableLogic） |

明细表 `biz_pick_list_detail`（多商品行）：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK auto | |
| pick_list_id | bigint | 主表ID（索引） |

> 主从表 CRUD 参照现有"工作要求主表 + 分配表 + 附件表"的主从模式（`WorkRequirement` / `WorkRequirementAssign` / `WorkRequirementAttachment`）。

## 1. 后端任务

### 任务 B1：数据库建表（主 + 明细）

- **验收**：编译通过；字段与表一致。

### 任务 B3：Mapper（主 + 明细）

  - `mapper/BizPickListMapper.java`：`extends BaseMapper<BizPickList>`。

- **验收**：编译通过。

### 任务 B4：DTO（主从结构）
- **文件**：
  - `dto/PickListSaveDTO.java`：`pickType`(NotNull, PICK/SUPPLY/RETURN)、`sourceSalesId`(可选，Q3)、`remark`(可选)、`details`(NotNull, `@Valid List<PickListDetailDTO>`, `@Size(min=1)`)。

  - `dto/PickListQueryDTO.java extends PageQuery`：`pickNo`、`pickType`、`status`、`goodsName`（模糊匹配明细商品名）、`startDate`/`endDate`。

- **验收**：jakarta validation 注解齐全（含嵌套 `@Valid`）；编译通过。

### 任务 B5：VO（主从结构）
- **文件**：
  - `vo/PickListVO.java`：单据头字段 + `List<PickListDetailVO> details` + `statusText`（状态文本）。
  - `vo/PickListDetailVO.java`：明细行字段。
- **验收**：编译通过。

### 任务 B6：编号生成器
- **文件**：`back/src/main/java/org/example/back/common/util/CodeGenerator.java`
- **内容**：新增 `public static String pickListNo()`，参照 `salesNo()` 规则（如 `PL` + 日期 + 序列）。
- **验收**：单测或手动调用返回非空且唯一。

### 任务 B7：Service
- **文件**：`back/src/main/java/org/example/back/service/PickListService.java`
- **内容**：参照 `SalesService.java` + 工作要求主从表模式，实现以下方法：
  - `page(PickListQueryDTO)`：权限分流（申请人看本人 / 销售 admin 看本部门 / 仓储看全部 / 超管看全部）；`goodsName` 模糊匹配需联查明细表（先查明细匹配的 pickListId 集合，再并入主表查询）。分页返回主表，每条主表附带明细列表。
  - `getById(Long id)`：详情（主表 + 明细），校验访问权限。
  - `create(PickListSaveDTO)`：建申请单，status=1，**不扣库存**。插入主表后批量插入明细（`sortNo` 自动补序）。校验所有明细商品启用状态。
  - `issue(Long id)`：仓储发料。按 `pick_type` 区分：
    - PICK / SUPPLY：明细逐条 `decreaseStock`，**任一缺料则整单回滚**（事务保护），全成功后 status→2。
    - RETURN：明细逐条 `increaseStock`（回流入库），status→2。
    - 记录 operator/operation_time。
  - `confirm(Long id)`：领料人确认收货，校验"本人申请"，status→3，记录 confirm_time。
  - `reject(Long id, PickListRejectDTO)`：仓储驳回，status→4，记录 reject_reason（不涉及库存）。
  - `delete(Long id)`：撤销待发料申请（仅 status=1 且本人可撤），级联删除明细，不涉及库存。
  - 私有 `increaseStock/decreaseStock`：参照 `SalesService` 第 310/320 行实现（含库存不足校验）。
- **权限**：
  - 申请：`requireAnyDeptAdminOrSuperAdmin(DEPT_SALES, DEPT_WAREHOUSE)`（Q1 确认）。
  - 发料/驳回：`requireDeptAdminOrSuperAdmin(DEPT_WAREHOUSE)`。
  - 确认：校验当前用户 = applicantId。
- **事务**：`@Transactional(rollbackFor = Exception.class)` 覆盖 create/issue/reject/delete。
- **验收**：各状态流转正确；PICK/SUPPLY 缺料整单回滚；RETURN 入库正确；权限边界生效。

### 任务 B8：Controller
- **文件**：`back/src/main/java/org/example/back/controller/PickListController.java`
- **内容**：参照 `SalesController.java`，`@RequestMapping("/business/pick-lists")`：
  - `GET /page` — 分页
  - `GET /{id}` — 详情
  - `POST` — 申请建单（`@PreventDuplicateSubmit`）
  - `PUT /{id}/issue` — 发料（`@RequireAdmin` + `@AuditLog`）
  - `PUT /{id}/confirm` — 确认收货
  - `PUT /{id}/reject` — 驳回（`@RequireAdmin` + `@AuditLog`）
  - `DELETE /{id}` — 撤销申请
- **验收**：Knife4j（`/api/doc.html`）可见全部接口；权限注解生效。

### 任务 B9：编译与启动验证
- **命令**：`cd back && ./mvnw -q -DskipTests compile`，然后重启后端。
- **验收**：编译无错；后端启动日志正常；`/api/business/pick-lists/page` 返回 401（未登录）或 200（已登录）。

## 4. 任务依赖与建议顺序

```
B1 建表 → B2 Entity → B3 Mapper → B4 DTO → B5 VO → B6 编号 → B7 Service → B8 Controller → B9 编译验证

F1 API → F2 View → F3 Router → F4 Permission → F5 联调
```

后端 B1–B9 完成后即可联调前端。建议先做 B1–B8 跑通接口，再做前端。

## 5. 验收标准（模块级）

1. 领料/补料支持多商品明细；申请后库存不扣，发料时才扣减；**任一明细缺料则整单回滚**。
2. 退料发料时库存回流入库（退料也走 待发料→已发料→已完成）。
3. 状态流转：待发料→已发料→已完成；驳回→已驳回；撤销仅限待发料且本人。
4. 权限：销售/仓储 admin 可申请；仓储发料/驳回；申请人确认本人单据；列表按角色过滤（本人/本部门/全部）。
5. 操作有审计日志（`@AuditLog`）；防重复提交（`@PreventDuplicateSubmit`）。
6. Knife4j 接口文档齐全；前端联调全流程（含多商品明细增删）通过。
